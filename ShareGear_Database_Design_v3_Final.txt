ShareGear - Final Database Design v3.0
Final merged schema after v2.0 tickets/forms update and agreed gap decisions
1. Purpose
This document is the final database design for ShareGear after applying the agreed database gap decisions to Database Design v2.0. The v2.0 database is used as the base because it already replaced Complaints with Tickets, added TicketMessages and TicketAttachments, added EmployeeProfiles, and added the Job Completion and Field Verification evidence forms.
2. Final Scope Rules
Apply every agreed GAP marked as apply in full unless an explicit exception was documented.
Keep the v2 ticket system and the two evidence form workflows.
Do not include removed or rejected objects in the final logical schema.
Migration scripts may temporarily keep deprecated columns only for backfill. The final logical schema reflects the target state.
3. Final Table Count
Final target schema contains 41 tables.
4. Removed / Not Applied Items
Item
Final Decision
Offers table
Removed by GAP-017. Offer negotiation is not part of the final SQL schema.
AuditLogs table
Removed by GAP-026. SQL table is not included in the final schema.
Complaints table
Already removed in v2 and replaced by Tickets.
Bookings.IsPriorityBooking
Removed by GAP-003 decision.
Priority admin acknowledgment gate
Not applied. Do not add PrioritySurchargeRate, PriorityAcknowledgedAt, PriorityAcknowledgedByAdminId, or PendingAdminPriorityAck.
Emergency override fields
Not added under GAP-001: IsEmergencyCancellation, EmergencyOverrideStatus, EmergencyOverrideAdminId, EmergencyOverrideDecidedAt.
Customer holdout fields
GAP-006 not applied. Do not add CustomerHoldoutElected or ExtendedProviderDeadline.
UserChatViolations table and Messages.ViolationType
GAP-013 not applied. Keep old chat-blocking design without this table.
ServiceListings.AvailabilityHours
Replaced by ServiceListingAvailability after migration/backfill.
Messages.IsRead and Messages.ReadAt
Replaced by ConversationReadReceipts. Can remain temporarily during migration only as denormalized cache.

10. Service-Layer Requirements
EscrowService must validate BookingTimeAdjustmentRequests.CustomerAcknowledgedAt is not null before modifying EscrowRecords.GrossAmount.
EscrowService must store EscrowRecords.AdjustmentRequestId when a time-adjustment request changes escrow.
Escrow release must use rowversion concurrency on EscrowRecords.Timestamp. A concurrent release attempt must fail and be logged.
Bookings updates must use Bookings.Timestamp as the EF Core concurrency token.
CancellationRefundPct must be stored at cancellation time and must not be recomputed later for disputes.
Out-of-zone surcharge acceptance must set OutOfZoneSurchargeAcceptedAt before payment proceeds.
TicketService must compute AdminResponseDeadline at ticket creation using the business-day SLA calendar.
Conversation unread counts must be derived from ConversationReadReceipts rather than writing Messages.IsRead per message.
Qdrant synchronization must be driven through DomainEventQueue, not an in-memory event bus.
ServiceListingAvailability is the source of truth for operational hours after migration backfill.
Full-text duplicate listing checks must use SQL Server full-text indexes and a PlatformSettings threshold.
AI decisions and admin overrides must be written to AiInteractionLogs with 90-day retention cleanup.
